# 010. HttpServer 生产可用性修复

## 概述

用户反馈 HTTP 服务器运行数天后出现崩溃、无响应等故障。经完整审查 14 个类的源码，发现 **3 个致命级 BUG + 5 个高危级 BUG + 7 个中危级问题**，其中致命级问题直接导致"运行几天就崩溃/无响应"的现象。

## 问题总览

| 级别 | 编号 | 问题 | 根因 | 影响 |
|------|------|------|------|------|
| 致命 | BUG-1 | 连接池索引漂移 | Collection.Remove 导致后续索引前移 | 数据发错客户端、操作已释放对象、GPF 崩溃 |

## 致命级问题详细分析

### BUG-1: 连接池 Collection 索引漂移

**症状**: 运行一段时间后数据发错客户端，或访问已释放对象导致 GPF 崩溃。

**根因**: `m_cConnPool` 使用 `Collection` 存储连接，每个 `cClientCallback.Index` 记录自身在 Collection 中的位置。当某个连接断开执行 `m_cConnPool.Remove Index` 时，后续所有项的索引自动前移，但 `oCallback.Index` 仍是旧值。

**示例**:
```
连接池: [A(1), B(2), C(3), D(4)]
B 断开 → Remove 2
连接池: [A(1), C(2), D(3)]   ← C 和 D 的索引变了
但 C.Index 仍为 2，访问 m_cConnPool(2) 得到 C 没问题
但 D.Index 仍为 3，访问 m_cConnPool(3) 得到 D 还是没问题？
错！下次 E 连接：m_cConnPool(5) 而中间空洞 4 不存在
更糟的是：如果 A 断开 Remove 1，B 已不在，C 从 2→1，D 从 3→2
但 C.Index=2, D.Index=3，此时 m_cConnPool(3) 越界或取到错误对象！
```

**修复**: 将 `m_cConnPool` 从 `Collection` 改为 `Scripting.Dictionary`，以 `hSocket`（套接字句柄）为 Key。hSocket 是操作系统分配的唯一标识，不会因其他连接的加入/移除而变化。

```vb
' 修复前
Private m_cConnPool As New Collection          ' 顺序索引
oCallback.Index = m_cConnPool.Count            ' 位置作为索引
m_cConnPool.Add oCallback                      ' Add 到末尾
Set oCallback = m_cConnPool.Item(Index)        ' 按位置查找
m_cConnPool.Remove Index                       ' 按位置移除

' 修复后
Private m_cConnPool As New Scripting.Dictionary ' hSocket 为 Key
oCallback.hSocket = sockHandle                  ' 唯一标识
m_cConnPool.Add sockHandle, oCallback           ' 按 hSocket 存储
Set oCallback = m_cConnPool.Item(hSocket)       ' 按 hSocket 查找
m_cConnPool.Remove hSocket                      ' 按 hSocket 移除
```

### BUG-2: 僵尸连接无限累积

**症状**: HTTP 服务器运行几天后内存持续增长，最终 OOM 崩溃。

**根因**: 当客户端异常断开（如断网、杀进程），不发送 TCP FIN 包，服务端的 `Socket_CloseSck` 事件永远不触发。`cClientCallback` 及其持有的 `Context/Request/Response/Session/Socket` 全部驻留内存。

**循环引用链**使得即使外部引用断开，COM 引用计数也不归零：
```
cHttpServer ← m_cConnPool ← cClientCallback ← Parent ──→ cHttpServer
cClientCallback ← Context ← Response ← fClient ──→ cClientCallback
```

只有 `Release()` 方法能断开这些循环引用，但 `Release()` 仅在 `OnCloseSck` 中调用——僵尸连接永远不会触发。

**修复**:
1. 新增 `LastActivity` 字段，每次数据到达时更新
2. 新增 `IdleTimeoutSeconds` 属性（默认 120 秒）
3. 新增 `CleanupIdleConnections()` 方法，遍历连接池释放超时连接
4. 在 `OnDataArrival` 中以 1% 概率触发清理

### BUG-3: 请求体无大小限制

**症状**: 恶意客户端发送超大 `Content-Length` 直接撑爆服务器内存。

**根因**: 服务器无脑接收数据，`AppendToBuffer` 不断 `ReDim Preserve` 扩大缓冲区。

**修复**:
1. 新增 `MaxRequestSize` 属性（默认 10MB）
2. `OnDataArrival` 中检查缓冲区 + 新数据是否超限，超限则断开
3. `ContainsCompleteRequest` 中检查 Content-Length 是否超限
4. 新增 `State413` 响应

## 高危级问题详细分析

### BUG-4: recvBuffer 永不缩容

**根因**: 长连接上处理完 10MB 上传后，`recvBuffer()` 数组保持已扩展的大小（10MB），后续小请求也占用 10MB 缓冲区。

**修复**: 在 `RemoveFromBuffer` 中，当缓冲区清空时 `Erase recvBuffer`；当缓冲区容量超过 64KB 且使用率低于 1/4 时，`ReDim Preserve` 缩容。

### BUG-5: 循环引用链（与 BUG-2 协同）

**修复**: `Release()` 中彻底释放所有引用并 `Erase recvBuffer`、重置 `hSocket = 0`。`Class_Terminate` 中调用 `StopMe` 防止意外销毁时泄漏。

### BUG-6: FindCRLFCRLF 返回 0 歧义

**根因**: CRLFCRLF 出现在 buffer 位置 0 时（如 `"\r\n\r\n"` 紧跟数据），`FindCRLFCRLF` 返回 0；未找到时也返回 0。调用者将 0 当"未找到"处理，导致这种合法请求永远无法解析。

**修复**: 未找到时返回 `-1`，`ContainsCompleteRequest` 和 `ExtractOneRequest` 同步更新判断。

### BUG-7: 无最大连接数限制

**修复**: 新增 `MaxConnections` 属性（默认 1000），`ConnectionRequest` 事件中超限直接 `Disconnect = True`。

### BUG-8: SendData 无错误处理

**根因**: 客户端已断开连接时，`Socket.SendData` 抛出异常，导致未处理的错误传播。

**修复**: `SendBodyByte` 和 `SendHeader` 中 `Socket.SendData` 加 `On Error Resume Next` 保护。同时新增 `m_HasSent` 标记防止重复发送。

## 内置定时清理机制

服务器启动后自动创建 `cTimer` 定时器（依赖 `ToolsTimer.bas` 模块），默认每 30 秒触发一次，执行：

1. `CleanupIdleConnections` — 清理超过 `IdleTimeoutSeconds` 的僵尸连接
2. `CleanupExpiredSessions` — 清理过期的 Session

用户可通过 `CleanupTimerInterval` 属性调整间隔（毫秒），设置后若定时器已运行会自动重启生效。服务器停止时定时器自动销毁。

此机制彻底解决了"服务器空闲时僵尸连接无法清理"的问题，无需外部定时器。

## 第二轮内存泄漏修复（2026-06-13 压测复现）

### 现象

修复第一轮 BUG 后，压测发现内存仍以 ~10MB/s 速度增长：10 连接、900 QPS，339 秒后内存从初始值涨到 1.8GB。

### 根因

`ProcessHttpRequest` **每次请求**创建约 18+ COM 对象（Request + 5 个 Dictionary、Response + Header、Cookies + 2 个 Dictionary、Session + Data 等），请求处理完毕后这些对象仅通过 VB6 COM 引用计数等待自动回收。但在高并发（900+ QPS）下：

1. **Response.fClient → oCallback 循环引用**：`oCallback → Context → Response → fClient → oCallback`，即使下次请求覆盖 `Context.Response`，旧 Response 仍因 fClient 引用保持存活
2. **Request 内部大对象未及时清理**：5 个 Dictionary + PathInfoList + pvRawBodyBin 字节数组
3. **VB6 引用计数在高频创建/销毁下回收滞后**：每秒 900 次创建 × 18 对象 = 16200 COM 对象/秒来不及回收

### 修复

| 文件 | 修改 |
|------|------|
| `cHttpServer.cls` | ProcessHttpRequest 的三个退出点（State413、Options、正常出口）+ 错误处理 EH 中，在 Exit Sub 前调用 `Context.ReleaseRequest` + `Request.ClearRequest` 显式清理 |
| `cHttpServerContext.cls` | 新增 `ReleaseRequest()` 方法：断开 Response.fClient 循环引用、清理 Request/Response/Cookies/Session，保留 SSE/UserData/TimeUse 等长生命周期对象 |
| `cHttpServerRequest.cls` | 新增 `ClearRequest()` 方法：RemoveAll 所有内部 Dictionary、Erase pvRawBodyBin、清空大字符串；新增 Class_Terminate 保险清理 |
| `cHttpServerResponse.cls` | 新增 Class_Terminate：Set Client = Nothing、Header.RemoveAll、释放 JsonInst |

### 清理链路

```
ProcessHttpRequest 请求处理完毕
  │
  ├─ Context.ReleaseRequest()
  │    ├─ Response.fClient = Nothing     ← 断开最关键的循环引用
  │    ├─ Request.ClearRequest()         ← 清空 Request 内部大对象
  │    ├─ Set Request = Nothing
  │    ├─ Set Response = Nothing
  │    ├─ Cookies.Clear + Set Cookies = Nothing
  │    ├─ Set Session = Nothing
  │    └─ UserData.RemoveAll
  │
  └─ Set response = Nothing: Set Request = Nothing  ← 释放局部变量引用
```

**最低升级要求**: 替换 cHttpServer.cls + cHttpServerContext.cls + cHttpServerRequest.cls + cHttpServerResponse.cls 四个文件

## 第三轮 Session 累积 + As New 全量修复（2026-06-13）

### 现象

第二轮修复后内存泄漏速度仍在约 5-10MB/s（10 连接、900 QPS 压测）。

### 根因分析

经深入审计，发现两个关键根因：

#### 根因 1: Session 自动生成 sessionID 导致空 Session 大量累积

`cHttpServerSession.sessionID` Property Get 在 `pvSessionID` 为空时**自动生成 GUID**。而 `ProcessHttpRequest` 中通过 `If Session.sessionID <> "" Then` 判断是否存储 Session——这导致**每个无 Cookie 的请求**都会：

1. 创建新的 `cHttpServerSession`
2. `sessionID` Property Get 触发自动生成 GUID
3. 条件 `Session.sessionID <> ""` 始终为 True
4. `m_Sessions(Session.sessionID) = Session` 将空 Session 永久存入字典

在 900 QPS 无 Cookie 压测下，20 分钟内累积 `900 × 60 × 20 = 1,080,000` 个空 Session，每个含 `Scripting.Dictionary` COM 对象，占内存约 1-2GB。

#### 根因 2: 全量 `As New` 隐式创建阻止显式释放

VB6 的 `Dim X As New ClassName` 语法：`Set X = Nothing` 后，下次访问 X 会**自动重建实例**，对象永远无法真正销毁。此问题存在于所有 14 个类文件中：

- **cHttpServerSession.cls**: `Dim Data As New Scripting.Dictionary` — 每个 Session 的内部字典永远无法释放
- **cHttpServer.cls**: `m_cConnPool As New`、`m_Sessions As New`、`SSE As New`、`Router As New` 等 9 个字段
- **cSSE.cls**: `Json As New`、`Clients As New`、`Data As New`、`MapClients As New` 等 7 个字段
- **cHttpServerSvr.cls**: `rootFiles As New`、`rootDirs As New`、`Fso As New`、`m_DefaultDocuments As New` 4 个字段
- **cHttpServerRouter.cls**: `Controllers As New`、6 个方法字典 `As New`、`DicGroup As New` 8 个字段
- **cHttpServerRouteBefore.cls**: `Dic As New`
- **cHttpServerRouterAfter.cls**: `Dic As New`

### 修复清单

#### 核心修复：Session 累积（消除主要泄漏源）

| 文件 | 修改 |
|------|------|
| `cHttpServerSession.cls` | **sessionID Property Get 不再自动生成 GUID**，仅返回 `pvSessionID`；新增 `HasID` 属性判断是否有实际数据；`Data` 改为延迟创建（EnsureData 辅助方法），空 Session 不分配 Dictionary；新增 `Release()` 方法；新增 `Class_Terminate` |
| `cHttpServer.cls` | **ProcessHttpRequest 中 `If Session.HasID Then` 替代 `If Session.sessionID <> "" Then`**，只有实际写入了数据的 Session 才会存储到 m_Sessions 和写入 Cookie；显式清理本地变量 `Set Session = Nothing`、`Set Cookies = Nothing` |

**效果**：无 Cookie 的压测请求不再创建空 Session 并存入 m_Sessions，空 Session 在请求结束后立即被 VB6 COM 引用计数回收。

#### As New 全量转换

| 文件 | As New 字段数 | 修改 |
|------|-------------|------|
| `cHttpServer.cls` | 9 | 所有字段改 `As`，Class_Initialize 中显式 `Set = New`，Class_Terminate 中显式 `Set = Nothing`；本地变量 `Dim Request As New` → `Dim Request As` + `Set Request = New` |
| `cHttpServerSession.cls` | 1 | `Data As New` → `Dim Data As` + EnsureData 延迟创建 |
| `cSSE.cls` | 7 | 全部改 `As` + Class_Initialize 显式创建 + Class_Terminate 清理；BUG-9 修复：Entry 失败时清理已设置的 SSE 引用 |
| `cHttpServerSvr.cls` | 4 | 全部改 `As` + Class_Initialize 显式创建 + Class_Terminate 清理 |
| `cHttpServerRouter.cls` | 8 | 全部改 `As` + Class_Initialize 显式创建 + Class_Terminate 清理 |
| `cHttpServerRouteBefore.cls` | 1 | 改 `As` + Class_Initialize + Class_Terminate |
| `cHttpServerRouterAfter.cls` | 1 | 改 `As` + Class_Initialize + Class_Terminate |

### 新增 API

| 属性/方法 | 类型 | 说明 |
|-----------|------|------|
| `cHttpServerSession.HasID` | Property Get (Boolean) | 判断 Session 是否已有 sessionID（即是否有实际数据被写入），用于替代直接判断 sessionID 是否为空 |
| `cHttpServerSession.Release()` | Friend Sub | 显式释放 Data Dictionary，清空 sessionID，用于连接关闭时的彻底清理 |

### 清理链路（更新）

```
ProcessHttpRequest 请求处理完毕
  │
  ├─ Context.ReleaseRequest()
  │    ├─ Response.fClient = Nothing     ← 断开最关键的循环引用
  │    ├─ Request.ClearRequest()         ← 清空 Request 内部大对象
  │    ├─ Set Request = Nothing
  │    ├─ Set Response = Nothing
  │    ├─ Cookies.Clear + Set Cookies = Nothing
  │    ├─ Set Session = Nothing
  │    └─ UserData.RemoveAll
  │
  ├─ Set Session = Nothing               ← 释放本地变量引用（不再等 Sub 退出）

  └─ Set response = Nothing: Set Request = Nothing
```

**新增逻辑**：只有 `Session.HasID = True` 时，Session 才会：
- 被存入 `m_Sessions` 字典
- 被写入响应 Cookie
- 被持久化到文件/数据库

## 第四轮 方法级 As New + SSE BUG-15/BUG-10 修复（2026-06-13）

### 修改内容

#### 方法级 `As New` 局部变量全量转换

上一轮修复了类级 `As New` 字段，但方法体内的 `Dim X As New` 局部变量仍有 11 处未处理。虽然局部变量出作用域后通常由 COM 引用计数回收，但 `As New` 的自动重建机制在异常路径下可能阻止回收，且与用户"彻底 100% 搞定"的要求一致。

| 文件 | 修改数量 | 具体变更 |
|------|---------|---------|
| `cHttpServer.cls` | 4 处 | `Dim CI As New cHttpServerClientInfo` → 显式；`Dim keysToClose As New Collection` → 显式；`Dim keysToRemove As New Collection` → 显式；`Dim sess As New cHttpServerSession` → 显式 |
| `cHttpServerSession.cls` | 4 处 | Serialize 中 `Dim Json As New cJson` + `SessionData As New Dictionary` + `DataCopy As New Dictionary` → 全部显式；Deserialize 中 `Dim Json As New cJson` → 显式 |
| `cHttpServerResponse.cls` | 1 处 | Json 方法中 `Dim Dic As New Scripting.Dictionary` → 显式 |
| `cHttpServerCookies.cls` | 1 处 | Cookie Property Get 中 `Dim CK As New cHttpServerCookieAttr` → 显式 |
| `cHttpServerRouter.cls` | 1 处 | Add 方法中 `Dim item As New cHttpServerRouteItem` → 显式 |

#### BUG-15 修复：SSE CloseClient 不关闭 Socket

**原实现问题**：`CloseClient` 方法存在严重的结构缺陷：
- 通过 `hSocket` 参数调用时（最常见方式），仅检查客户端是否存在，然后 `GoTo QH` 退出——**不做任何清理**
- 通过 `ToUser().CloseClient` 调用时（GoSub Start 路径），清理字典但不关闭 Socket

**后果**：所有主动关闭的 SSE 客户端，Socket 永远不关闭，cClientCallback 永远不释放，内存持续泄漏。

**修复**：重构 CloseClient 为统一清理路径：
1. 触发 OnClose 用户事件
2. 清除 SSE 上下文引用（`Context.SSE = Nothing`）
3. 从 Clients 字典移除（防止重入）
4. 清理用户/分组映射
5. **调用 `Client.Socket.CloseSck`** 关闭底层 Socket，触发 cClientCallback 完整清理流程

#### BUG-10 修复：SSE.Data 共享可变状态重入问题

**原实现问题**：`Send` 方法使用类级共享 `Data` 字典，`SendData` 过程中可能触发 Winsock 消息泵导致重入调用，共享 Data 内容被覆盖。

**修复**：`Send` 方法在编码前先将 `Data` 快照到局部 `SendData` Dictionary，然后立即 `Data.RemoveAll`。编码和发送使用局部安全副本。`SendPack` 无需修改，因为它调用 `Send` 时共享 Data 已在快照后被清空。
